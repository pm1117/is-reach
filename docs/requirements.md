# is-reach 要件定義書

- ステータス: **草案（人間の承認待ち）**
- フェーズ: 1（要件定義）
- 本書の承認をもってフェーズ2（基本設計 `docs/basic-design.md`）へ進む。承認前に設計・実装には着手しない。
- 本書では確定した事項を「**決定**」、未確定の前提を「**仮置き**」と明記して区別する。

## 1. 概要・目的

is-reach は、インサイドセールス（IS）チーム向けの SaaS プロダクトである。以下の一連の業務を支援する:

1. **スクリーニング**: 条件検索により見込みのありそうな企業リストを自動抽出する
2. **深掘り**: 人が選んだ企業について公開情報を収集・分析し、企業ごとの調書（ドシエ）を作る
3. **メッセージ生成**: ドシエをもとに、問い合わせフォームに送るパーソナライズされたメッセージ案を生成する

送信自体はツールの責務外であり、**人間が内容を確認し、人間がフォームから送信する**（決定 A4）。ツールの価値は「送る相手を見つけ、調べ、書く」までの時間を大幅に短縮することにある。

## 2. 用語定義

| 用語 | 意味 |
|------|------|
| スクリーニング | 企業属性・公開シグナルの条件検索により、企業プールから見込み企業リスト（100〜500社規模）を抽出する工程 |
| 公開シグナル | 求人情報・技術ブログ・プレスリリースなど、企業の「今」を示す公開情報。検索インテントデータの MVP における代替 |
| 深掘り | 人が選択した企業についてオンデマンドで HP・公開記事を収集し、LLM で分析する工程 |
| ドシエ | 深掘りの成果物。事業サマリ・推定課題・自社サービスとの接続点・根拠 URL を構造化した企業調書 |
| テンプレート | テナントが登録するメッセージの骨子（自社紹介・CTA・トーン）。LLM はパーソナライズ部分のみを埋める |
| テナント | is-reach を契約して利用する企業（顧客）。データはテナント間で完全に分離される |

## 3. スコープ

### 3.1 MVP に含む（決定）

- スクリーニング検索（企業属性 + 公開シグナルによる条件検索・リスト保存）
- 深掘り実行とドシエ閲覧（選択企業のみ・オンデマンド）
- メッセージ生成・編集・コピーと送信ステータス管理（テンプレート管理を含む）
- テナント・ユーザー管理（招待、管理者/メンバー権限）
- 非機能要件: プロンプトインジェクション対策、スクレイピング節度、PII 管理、監査ログ

### 3.2 スコープ外（決定）

- **問い合わせフォームの自動送信・自動操作**（人手確認 + 人が送信。スパム規制・サイト利用規約リスクのため）
- X / LinkedIn 等 SNS の直接クロール（利用規約違反・法務リスクのため）
- 海外企業・多言語対応（日本企業・日本語前提）

### 3.3 将来拡張（仮置き — MVP では実装しないが設計時に閉ざさない）

- 外部インテントデータ（検索行動データ）プロバイダとの連携
- X / LinkedIn の**公式 API 等の合法ルート**による経営層の発信取得
- 受注確度スコアリング・優先順位付けの自動化（大手競合との差別化を見て判断）
- CRM 連携（Salesforce / HubSpot 等）
- 課金機構

## 4. ユーザーと提供形態

- **決定 A1**: 最初から SaaS として複数企業に外販する。テナント分離・認証・権限管理は初期要件。
- **決定 A2**: エンドユーザーはテナント企業の IS 担当者。アプローチ先は日本企業・業種汎用（日本語サイト前提）。
- **仮置き**: 認証はメール + パスワード（将来 SSO 対応）。課金機構は MVP スコープ外。

## 5. 機能要件

### F1. スクリーニング（企業リスト抽出）— 決定 A3 / A3-1

条件検索により企業プールから見込み企業リストを抽出する。

- 検索条件の例: 従業員数、業種、地域、公開シグナル（「React の求人を出している」「特定技術のブログ記事がある」等）
- データの性質: **構造化・広く浅く**。定期バッチで事前収集・キャッシュし、検索は即時に返す
- LLM は原則使用しない（フィルタとルールベースのスコアリング）
- 抽出規模: 1 回あたり 100〜500 社
- 検索インテント（「今検索している」）は MVP では公開シグナルで代替する（決定 A3-1）

受け入れ条件:
1. 企業属性 + 公開シグナルの組み合わせ条件で検索し、結果をリストとして保存できる
2. 検索結果には各企業のマッチ根拠（どのシグナルに該当したか）が表示される

### F2. 深掘り収集 — 決定 A3 / A3-2

ユーザーがリストから選択した企業についてのみ、オンデマンドで公開情報を収集する。

- 収集対象: 企業公式サイト（事業内容・ニュース・採用）、経営層の**公開記事**（インタビュー・登壇情報・役員紹介ページ）
- データの性質: **非構造・狭く深く**。1 社あたりのコストは高くてよい（対象は選択された数十社のみ）
- SNS（X / LinkedIn）の直接クロールは行わない（決定 A3-2）
- 収集した外部コンテンツはすべて**信頼境界外のデータ**として扱う（→ 6.1）

受け入れ条件:
1. リストから選択した企業に対して深掘りを実行でき、進捗・完了・失敗が画面で確認できる
2. 収集結果にはすべて出典 URL が保持される

### F3. 分析（ドシエ生成）— 決定 B1

深掘りで収集したデータを LLM で分析し、構造化されたドシエを出力する。

- 出力項目: 事業サマリ / 推定課題 / 自社サービスとの接続点（フック）/ 各項目の根拠 URL
- 受注確度スコアリングは MVP では行わない（将来検討・仮置き）

受け入れ条件:
1. ドシエの各主張（推定課題・接続点）に根拠 URL が紐づいている
2. 根拠が取れない項目は「根拠なし」と明示され、事実のように表示されない

### F4. メッセージ生成 — 決定 B2

テンプレート骨子 + パーソナライズ埋め込み方式で問い合わせメッセージ案を生成する。

- テナントはテンプレート（自社紹介・CTA・トーン・文字数制約）を登録できる
- LLM はパーソナライズ部分（冒頭の接点・課題への言及）のみを生成し、テンプレ骨子に埋め込む
- 生成結果はユーザーが編集でき、コピーしてフォームに貼り付けて使う

受け入れ条件:
1. テンプレートを登録・編集でき、生成時に選択できる
2. 生成されたメッセージを編集・コピーできる
3. 生成文はテンプレートの骨子（自社紹介・CTA）を欠落させない

### F5. 送信ステータス管理 — 決定 A4

送信は人間が行うため、ツールは進捗管理のみ担う。

- 企業ごとのステータス（未着手 / 生成済み / 送信済み / 返信あり 等）を手動で更新・一覧できる

受け入れ条件:
1. 企業単位でステータスを更新でき、リスト画面で絞り込める

### F6. テナント・ユーザー管理 — 決定 A1 / B3

- テナントごとにデータを完全分離する（企業リスト・ドシエ・テンプレート・メッセージすべて）
- ユーザー招待、管理者 / メンバーの 2 権限（仮置き: 詳細なロール設計は基本設計で決める）

受け入れ条件:
1. あるテナントのユーザーが他テナントのデータに一切アクセスできない

## 6. 非機能要件

### 6.1 セキュリティ: プロンプトインジェクション対策（決定 C4 — 必須要件）

スクレイピング結果・外部コンテンツを LLM プロンプトに組み込むため、以下を必須とする:

- 外部コンテンツは**信頼境界の外**として扱い、システム指示と厳格に分離された「データ領域」としてのみ LLM に渡す
- 外部コンテンツ内のテキストが指示として解釈されない構造（区切り・エスケープ・ロール分離）を設計段階で定義する（→ 基本設計）
- 生成結果が外部コンテンツ由来の不正な指示に従っていないかの検証観点をレビュー必須項目とする

### 6.2 スクレイピング節度（仮置き C1）

- robots.txt を遵守する
- 同一ドメインへのアクセスにレート制限を設ける
- User-Agent を明示する
- 対象サイトに過剰な負荷を与えない（具体値は基本設計で定める）

### 6.3 個人情報（PII）・監査ログ（仮置き C2）

- 保存する個人情報は**公開情報由来**（役員名・公開記事）に限定する
- 個人情報を含むデータには必ず出典 URL を保持する
- 削除依頼に対応できる（企業・個人単位でのデータ削除）
- 監査ログ: 誰がいつ検索・深掘り・生成・コピーしたかを記録する

### 6.4 規模・性能（決定 C3 — 小規模スタート）

| 項目 | 基準値 |
|------|--------|
| テナント数 | 〜10 |
| ユーザー数 | 〜10 / テナント |
| 深掘り実行数 | 〜50 社 / 日 / テナント |
| スクリーニング検索 | 即時応答（事前収集したデータに対する検索のため） |

スケールは将来課題とするが、設計で閉ざさない。

## 7. 決定事項・仮置き一覧

### 決定

| # | 項目 | 内容 |
|---|------|------|
| A1 | 提供形態 | 最初から SaaS 外販。テナント分離・認証・権限が初期要件 |
| A2 | ターゲット | 日本企業・業種汎用 |
| A3 | パイプライン | スクリーニング（構造化・広く浅く・事前バッチ）と深掘り（非構造・狭く深く・オンデマンド）の 2 段階 |
| A3-1 | インテント | MVP は公開シグナル（求人・技術ブログ・プレスリリース）で代替 |
| A3-2 | 経営層情報 | 公開記事・自社サイトのみ。SNS 直接クロール禁止 |
| A4 | 送信 | 人手確認 + 人が送信。自動送信はスコープ外 |
| B1 | 分析出力 | 事業サマリ・推定課題・接続点・根拠 URL |
| B2 | 生成方式 | テンプレ骨子 + パーソナライズ埋め込み |
| B3 | 画面 | スクリーニング検索 / 深掘り+ドシエ / 生成・編集・コピー / テナント・ユーザー管理 |
| C3 | 規模 | 小規模スタート（6.4 の基準値） |
| C4 | セキュリティ | プロンプトインジェクション対策は必須要件 |

### 仮置き（承認時・基本設計時に確認）

| 項目 | 仮置き内容 |
|------|-----------|
| スクレイピング節度 | robots.txt 遵守・レート制限・UA 明示（具体値は基本設計） |
| PII・監査 | 公開情報由来のみ保存・出典保持・削除対応・監査ログ |
| 認証 | メール + パスワード（将来 SSO） |
| 課金 | MVP スコープ外 |
| 将来拡張 | インテントデータ連携 / X・LinkedIn 公式 API / スコアリング / CRM 連携 |
