# Software Architecture Skill

is-reach（インサイドセールス向け「データ取得 → 分析 → 問い合わせフォーム用メッセージ自動生成」）の設計を担当する。

## 原則

- **実装コードは書かない。** 成果物はすべて `docs/` 配下のドキュメント。
- フェーズゲートを守る: 要件定義 → 基本設計 → 詳細設計。前フェーズの人間承認なしに次へ進まない。
- 確定していない事項は「仮置き」と明記し、決定事項と区別する。

## フェーズごとの成果物と承認条件

| フェーズ | 成果物 | 次へ進む条件 |
|----------|--------|--------------|
| 要件定義 | `docs/requirements.md` | 人間が OK |
| 基本設計 | `docs/basic-design.md` | 人間が OK |
| 詳細設計 | `docs/design-detail.md` | 人間が OK |

## 基本設計で決めること

- モノレポ構成・パッケージ境界（初期案: `apps/web`, `apps/api`, `packages/crawler`, `packages/analysis`, `packages/prompt`, `packages/shared`）
- ドメインモデル・データモデル
- 取得 → 分析 → 生成のパイプラインと、クローラー ↔ LLM のデータ契約（型契約は `packages/shared` に集約）
- **セキュリティ境界**: 外部コンテンツ（スクレイピング結果）を信頼境界の外として扱い、LLM プロンプトへの注入経路を設計段階で隔離する

## 詳細設計で決めること

- API 契約
- プロンプト構成（システム / ユーザー / 外部データの分離）
- エラー状態・リトライ方針
- 非機能要件（レート制限、監査ログ、個人情報の扱い）

## 必須セキュリティ観点

スクレイピング結果や外部コンテンツを LLM プロンプトに組み込むため、**プロンプトインジェクション対策を設計の必須要件として扱う**。外部データは常に区切られたデータ領域として渡し、指示として解釈されない構造を設計に含めること。
